# 1941A - Rudolf And The Ticket

## Resumen del problema
Rudolf tiene monedas en dos bolsillos: el izquierdo y el derecho. Quiere comprar un billete para un evento. Para hacerlo, necesita seleccionar exactamente una moneda de su bolsillo izquierdo y exactamente una moneda de su bolsillo derecho. El billete tiene un costo máximo `k`. Rudolf quiere saber cuántas formas diferentes hay de seleccionar un par de monedas (una de cada bolsillo) de tal manera que la suma de sus valores no exceda `k`.

## Input
-   La primera línea contiene un solo entero `t` ($1 \le t \le 1000$), el número de casos de prueba.
-   Para cada caso de prueba:
    -   La primera línea contiene tres enteros `n`, `m`, y `k` ($1 \le n, m \le 100$; $1 \le k \le 2000$).
        -   `n`: la cantidad de monedas en el bolsillo izquierdo.
        -   `m`: la cantidad de monedas en el bolsillo derecho.
        -   `k`: el valor máximo que la suma de las dos monedas no debe exceder.
    -   La segunda línea contiene `n` enteros $a_1, a_2, \dots, a_n$ ($1 \le a_i \le 1000$), las denominaciones de las monedas en el bolsillo izquierdo.
    -   La tercera línea contiene `m` enteros $b_1, b_2, \dots, b_m$ ($1 \le b_j \le 1000$), las denominaciones de las monedas en el bolsillo derecho.

## Output
Para cada caso de prueba, imprime un solo entero: el número de pares de monedas que Rudolf puede elegir cuya suma no exceda `k`.

## Restricciones

| Variable     | Rango      | Notas                               |
| :----------- | :--------- | :---------------------------------- |
| t (casos)    | [1, 1000]  |                                     |
| n (monedas izq.) | [1, 100]   |                                     |
| m (monedas der.) | [1, 100]   |                                     |
| k (límite)   | [1, 2000]  |                                     |
| $a_i$ (valor moneda izq.) | [1, 1000]  |                                     |
| $b_j$ (valor moneda der.) | [1, 1000]  |                                     |

## Forma de resolverlo

### Enfoque Algorítmico (Iteración Exhaustiva de Pares)
El problema requiere contar todas las combinaciones de una moneda del bolsillo izquierdo y una del bolsillo derecho cuya suma no exceda un límite dado. Dado que las restricciones en el número de monedas (`n` y `m` hasta 100) son pequeñas, podemos utilizar un enfoque de fuerza bruta o iteración exhaustiva.

1.  **Iterar sobre todas las monedas del bolsillo izquierdo:** Recorrer cada moneda $a_i$ en la lista de denominaciones del bolsillo izquierdo.
2.  **Para cada moneda izquierda, iterar sobre todas las monedas del bolsillo derecho:** Para cada $a_i$, recorrer cada moneda $b_j$ en la lista de denominaciones del bolsillo derecho.
3.  **Verificar la condición:** En cada par $(a_i, b_j)$ formado, calcular su suma $a_i + b_j$.
4.  **Contar pares válidos:** Si la suma $a_i + b_j$ es menor o igual que el límite `k`, incrementar un contador de pares válidos.
5.  **Imprimir el resultado:** Después de revisar todas las combinaciones posibles, el valor final del contador será la respuesta.

Este método garantiza que se revisen todas las combinaciones posibles y se cuenten correctamente aquellas que cumplen la condición.

## Algoritmo
1.  Leer el número de casos de prueba `t`.
2.  Para cada caso de prueba:
    a.  Leer los enteros `n`, `m`, y `k`.
    b.  Leer las `n` denominaciones de monedas del bolsillo izquierdo en una lista/arreglo `A`.
    c.  Leer las `m` denominaciones de monedas del bolsillo derecho en una lista/arreglo `B`.
    d.  Inicializar una variable `contador_pares_validos` a 0.
    e.  Iterar con un bucle anidado:
        i.  Para cada `moneda_izquierda` en `A`:
            ii. Para cada `moneda_derecha` en `B`:
                iii. Si `moneda_izquierda + moneda_derecha <= k`:
                    Incrementar `contador_pares_validos` en 1.
    f.  Imprimir el valor de `contador_pares_validos`.

## Notas Adicionales
*   La simplicidad de las restricciones de `n` y `m` (hasta 100) hace que un enfoque de doble bucle sea perfectamente eficiente para este problema.
*   El problema no especifica si las denominaciones de las monedas son únicas o si están ordenadas, pero esto no afecta la lógica del algoritmo, ya que se revisan todas las combinaciones posibles independientemente de estos factores.

| Temporal | O(N * M)               | Se realizan N * M sumas y comparaciones en el peor caso. |
| Espacial | O(N + M)               | Se requiere espacio para almacenar las dos listas de denominaciones. |
